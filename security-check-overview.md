# Java SAST 도구 — 보안약점 점검 항목 및 탐지·수정 방식 요약

본 도구는 행정안전부 **「소프트웨어 보안약점 진단가이드(2021)」** 를 기준으로 Java 소스코드를 정적 분석(SAST)하여 총 **69개 보안약점**을 자동 탐지하고, 항목별 수정 방향을 제시합니다.

- **PART3 (DS-x.x)** — 분석·설계단계 보안설계 기준: **20개 항목**
- **PART4 (IV-x.x)** — 구현단계 보안약점 제거 기준: **49개 항목**

---

## 1. 자바 개발자를 위한 활용 가이드

이 도구는 "한 번 돌려보는 검사기"가 아니라, **개발 생명주기 전반에 걸쳐 시큐어 코딩을 습관화하는 도구**로 사용해야 합니다.

### 1.1 개발 단계별 활용 시점

| 시점 | 활용 방법 |
|---|---|
| **설계 단계** | PART3(DS-x.x) 항목을 보안설계 체크리스트로 활용합니다. 인증·접근통제·암호화·세션 설계가 가이드 기준을 충족하는지 코드 작성 전에 검토합니다. |
| **기능 구현 직후** | 새 기능(특히 외부 입력을 다루는 Controller·DAO·파일 처리 코드)을 작성한 뒤 해당 모듈을 분석하여 PART4(IV-x.x) 위반을 조기에 발견합니다. 수정 비용이 가장 낮은 시점입니다. |
| **코드 리뷰 / PR 전** | 분석 리포트를 첨부하여 CRITICAL·HIGH 항목이 0건임을 확인한 뒤 리뷰를 요청합니다. 리포트의 오염 흐름은 리뷰어가 취약 경로를 빠르게 이해하는 근거 자료가 됩니다. |
| **릴리스 전 정기 점검** | 전체 프로젝트를 분석하여 PDF 진단서를 보관하고, 이전 진단 대비 잔존·신규 약점 추이를 관리합니다. |

### 1.2 분석 결과를 읽고 조치하는 순서

1. **위험도순 우선 조치** — CRITICAL(IV-1.5 OS 명령어 삽입, IV-5.5 역직렬화) → HIGH(SQL 삽입, 경로 조작 등) → MEDIUM → LOW 순으로 처리합니다. LOW(디버그 코드, 주석 정보 등)도 릴리스 전에는 정리합니다.
2. **오염 흐름으로 원인 파악** — 리포트의 `Source → Propagator → Sink` 경로를 따라가며 외부 입력이 어디서 들어와(Source 라인) 어떤 변수를 거쳐(전파 변수) 어느 위험 API에 도달했는지(Sink 라인) 확인합니다. 단순히 Sink 라인만 고치는 것이 아니라 **입력 진입점부터 검증·바인딩을 적용**하는 것이 올바른 조치입니다.
3. **권고 수정 코드 적용** — 리포트의 권고 수정 코드는 탐지된 실제 변수명이 삽입되어 있으므로 거의 그대로 적용 가능합니다. 다만 트랜잭션·예외 처리 등 주변 코드 맥락에 맞게 다듬은 뒤 반영합니다.
4. **재분석으로 조치 검증** — 수정 후 동일 범위를 다시 분석하여 해당 항목이 사라졌는지 확인합니다. 수정 과정에서 새로운 약점이 생기지 않았는지도 함께 봅니다.
5. **근본 원리 학습** — 항목별 가이드 참조 페이지와 CWE·OWASP 링크를 읽고 같은 유형의 실수를 반복하지 않도록 합니다. 반복 검출되는 항목은 팀 코딩 컨벤션에 반영합니다.

### 1.3 오탐(False Positive)을 만났을 때

1. 먼저 **정말 오탐인지** 오염 흐름을 검토합니다. 입력 검증이 다른 계층(필터, 프레임워크)에서 수행되고 있는지 확인합니다.
2. 가능하면 억제보다 **Sanitizer 적용이 우선**입니다. `prepareStatement()`, HTML 인코딩, `getCanonicalPath()` 등 가이드가 인정하는 방식으로 코드를 바꾸면 탐지 자체가 사라집니다.
3. 구조적으로 안전함이 확인된 항목만 `sast-suppressions.json`에 억제 규칙을 등록하고, **억제 사유를 함께 기록**합니다. 테스트 코드 경로는 기본적으로 분석에서 제외됩니다.

### 1.4 하지 말아야 할 사용 방식

- 분석을 릴리스 직전에 한 번만 돌리는 방식 — 누적된 HIGH 항목을 한꺼번에 고치다 일정에 쫓겨 미루게 됩니다.
- 위험도 무시하고 검출 건수만 줄이기 — LOW 10건보다 CRITICAL 1건이 우선입니다.
- 검토 없이 일괄 suppression 등록 — 실제 취약점이 가려져 도구를 쓰는 의미가 없어집니다.
- 권고 수정 코드를 검증 없이 복사·붙여넣기 — 적용 후 반드시 빌드·테스트와 재분석으로 확인합니다.

> 실행 명령·화면 캡처 등 구체적인 조작 방법은 `user_manual.md`(또는 PDF/DOCX) 문서를 참고하세요.

## 2. 탐지 방식

모든 규칙은 `security-rules.json`에 정의되어 있으며, 규칙 성격에 따라 아래 엔진이 분석을 수행합니다.

| 탐지 방식 | 엔진 | 적용 규칙 수 | 동작 원리 |
|---|---|---|---|
| **오염 추적 (Taint Analysis)** | `TaintAnalysisEngine` | 14개 | 외부 입력(Source: `getParameter()`, `getenv()` 등) → 변수 전파(Propagation: 문자열 연결·대입) → 위험 API(Sink: `executeQuery()`, `exec()` 등) 도달 여부를 AST 기반 데이터플로우로 추적. 중간에 Sanitizer(`prepareStatement()`, `escapeHtml()` 등)를 거치면 오염 해제로 판단하여 오탐 방지 |
| **위험 패턴 매칭 (Pattern Analysis)** | `PatternAnalyzer` | 55개 | 규칙별 `dangerousPatterns` 정규식으로 취약한 API 호출·코딩 패턴(취약 암호 알고리즘, 하드코딩된 비밀번호, 빈 catch 블록 등)을 탐지 |
| **시퀀스 분석** | `SequenceAnalyzer` | IV-3.1 | 파일 검사 시점과 사용 시점 사이의 호출 순서를 분석하여 TOCTOU 경쟁조건 탐지 |
| **오탐 필터** | `FalsePositiveFilter` | 전체 | 테스트 경로 제외, Sanitizer 인식, 사용자 정의 억제 규칙(`sast-suppressions.json`)의 3-Layer 필터로 오탐 최소화 |

> AST 파싱은 JavaParser + SymbolSolver(타입 해석)를 사용하며, 메서드 단위 인트라-프로시저럴 데이터플로우로 분석합니다.

## 4. 수정 방향 제시 방식

탐지된 각 취약점에 대해 `RemediationService`가 **실제 코드 맥락이 반영된 수정안**을 자동 생성합니다.

1. **맞춤형 수정 코드** — 플레이스홀더가 아닌, AST에서 추출한 **실제 변수명이 삽입된** 수정 코드를 제시합니다.
   ```java
   // [수정] IV-1.1 SQL 삽입: PreparedStatement 사용
   String sql = "SELECT * FROM board WHERE b_gubun = ?";
   PreparedStatement pstmt = con.prepareStatement(sql);
   pstmt.setString(1, gubun);   // ← 탐지된 변수명 자동 삽입
   ```
2. **탐지 근거 설명** — 오염 흐름(Source 라인 → 전파 변수 → Sink 라인)을 한국어 문장으로 설명합니다.
3. **참고자료 연결** — 진단가이드 페이지(예: `PART4 제1절 1항 (p.180-193)`), CWE 번호, OWASP 참고 URL을 함께 제공합니다.
